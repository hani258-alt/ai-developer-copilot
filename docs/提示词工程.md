# 👨‍💻 提示词工程笔记 (代码项目实战版)

> **适用对象**：全栈开发者、AI 辅助编程爱好者
> **核心原则**：将 AI 视为**执行力极强的初级全栈工程师**，你是**架构师与代码审查员**。

## 一、 代码提示词的黄金结构 (CRISP-CODE)

针对写代码，通用的 CRISPE 框架需要微调。请严格遵循以下结构：

| 要素  | 英文    | 针对代码的特定含义    | 必填示例                                          |
| :---- | :------ | :-------------------- | :------------------------------------------------ |
| **C** | Context | **技术栈与环境**      | *使用 Python 3.11 + FastAPI + SQLAlchemy*         |
| **R** | Role    | **代码工种**          | *你是一名资深后端架构师，擅长高并发设计*          |
| **I** | Input   | **接口与数据定义**    | *输入是一个 JSON 对象，包含 `user_id` 字段*       |
| **S** | Steps   | **实现逻辑约束**      | *必须使用异步处理，必须包含类型注解 (Type Hints)* |
| **P** | Pattern | **设计模式/代码风格** | *请遵循 PEP8 规范，使用 Repository 模式*          |

### 📝 实战模板示例 (Python 后端接口)

```text
【角色】你是一名资深 Python 后端专家，代码风格严谨，注释清晰。
【任务】编写一个 FastAPI 的路由处理函数，用于用户注册。
【输入】接收 `username`, `email`, `password` 三个字段。
【约束】
1. 密码必须使用 `bcrypt` 进行哈希加密后再存储（伪代码即可，不需要连接真实数据库）。
2. 邮箱格式必须校验，若不符合返回 422 状态码。
3. 必须包含 Pydantic Model 定义和 Type Hints。
【输出】仅输出完整的 Python 代码块，不需要额外解释。
```







## 二、 全链路开发场景速查表 

将 AI 融入软件开发生命周期 (SDLC) 的每一步。

### 1. 需求分析与架构设计 🏗️

**场景**：将模糊的一句话需求转化为具体的开发任务拆解。
**提示词模板**：

> 我需要开发一个【项目名称：极简笔记应用】。技术栈是 React + Vite + TailwindCSS。
> 请帮我扮演**产品经理**，列出核心功能 MVP 清单。
> 然后切换为**前端架构师**，给出这个项目的**文件目录结构建议**。
> 最后切换为**技术负责人**，列出开发这些功能的前后顺序。

### 2. 编写单元函数 (Tool Function) 🔧

**场景**：写一个具体的工具类，如日期格式化、数据脱敏。
**提示词模板**：

> Act as a Python utility library author.
> 写一个函数 `mask_phone_number(phone: str) -> str`。
> 要求：保留前3位和后4位，中间用 `****` 代替。例如输入 `13812345678` 返回 `138****5678`。
> **必须包含 3 个 `doctest` 测试用例作为注释。**

### 3. 编写复杂业务逻辑 (Service Layer) 🧠

**场景**：写一个包含数据库查询、异常处理、事务逻辑的复杂模块。
**提示词模板**：

> 请使用 **思维链 (Chain of Thought)** 的方式写代码。
> 任务：实现一个转账函数 `transfer(from_id, to_id, amount)`。
> 步骤：
>
> 1. 先用注释写出伪代码逻辑步骤（Check, Debit, Credit, Commit）。
> 2. 在伪代码下方，将其翻译为实际的 `async` Python 代码（使用 SQLAlchemy）。
> 3. 注意：如果在第二步扣款失败，必须显式抛出 `InsufficientBalanceError` 自定义异常。

### 4. 代码解释与知识获取 (Reverse Engineering) 🔍

**场景**：接手了一个屎山代码，看不懂一个复杂的正则或算法。
**提示词模板**：

> 解释下面这段正则表达式的作用，并给出三个能匹配通过的字符串例子和三个不能通过的字符串例子。
> 【粘贴正则代码】

### 5. Bug 分析与修复 (Debugging) 🐛

**场景**：控制台报错，或者逻辑不符合预期。
**提示词模板 (必杀技)**：

> 我将提供一段代码和一段报错堆栈信息。
> **角色**：你是一名 SRE (站点可靠性工程师) 专家。
> **分析步骤**：
>
> 1. 指出导致错误的**直接原因**（第几行出了问题）。
> 2. 分析导致错误的**根本原因**（为什么会有这个数据）。
> 3. 给出**3种不同思路**的修复方案，并对比优缺点。
>    【粘贴代码】
>    【粘贴 Traceback 报错】

### 6. 代码重构与优化 (Refactoring) ✨

**场景**：优化性能差或难以维护的旧代码。
**提示词模板**：

> 重构下面这段 JavaScript 代码。
> **重构目标**：
>
> 1. 将 `var` 全部改为 `const/let`。
> 2. 将嵌套的回调地狱改为 `async/await` 风格。
> 3. 给复杂判断逻辑提取一个语义化的中间变量。
>    **注意**：**绝对不要改变原有的业务逻辑行为**。

### 7. 生成单元测试 (Unit Test) 🧪

**场景**：写完功能后，让 AI 帮你写最枯燥的测试用例。
**提示词模板**：

> 为下面这个 Python 函数生成 **Pytest** 单元测试代码。
> **覆盖要求**：
>
> - 正常路径 (Happy Path)
> - 边界情况 (Edge Cases: 空列表、极大值)
> - 异常抛出 (Exception)
>   **要求**：请使用 `pytest.fixture` 来管理测试数据。

## 三、 进阶技巧：上下文工程与 RAG

在大型代码库中使用 AI，必须解决**上下文长度限制**。

| 技巧             | 具体操作                                       | 工具推荐                                                 |
| :--------------- | :--------------------------------------------- | :------------------------------------------------------- |
| **文件锚定**     | 告诉 AI 当前引用的具体文件名。                 | 例如：*请参考 `@src/utils/auth.js` 的风格修改当前文件。* |
| **依赖注入**     | 粘贴 `package.json` 或 `requirements.txt`。    | 告诉 AI 库的版本，避免生成不兼容 API 的代码。            |
| **AST 切割**     | 手动只粘贴当前需要修改的**函数体**。           | 删除无关的 import 和类属性，节省 Token。                 |
| **Prompt Cache** | 将项目规范写在一个长文件中，每次对话开头引用。 | 例如：*后续所有代码请遵循【项目编码规范】。*             |

# 专业名词注释：

## 1.伪代码

**伪代码**是一种介于自然语言和编程语言之间的算法描述方式。它不遵循特定编程语言的严格语法，而是用简洁、易读的语句描述程序的逻辑流程，便于人类理解算法思路，也常用于教学和设计阶段。

**特点**：

- 无需纠结具体语法（如变量声明、括号、分号等）
- 使用常见的控制结构（如 `if-else`、`for`、`while`）
- 可结合自然语言（如“交换A和B的值”）
- 重点在于表达**做什么**，而非**如何用某语言写**

**示例**：
计算两个整数的最大公约数（欧几里得算法）：

```text
输入 a, b
while b ≠ 0:
    temp = a % b
    a = b
    b = temp
输出 a
```



伪代码没有统一标准，只要清晰、无歧义即可。



## 2.思维链

**思维链**（Chain of Thought，简称 CoT）是一种提示或推理技术，主要应用于大语言模型中。它引导模型在给出最终答案之前，先输出一系列**中间推理步骤**，将复杂问题分解为多个更简单的子问题，从而提升模型在逻辑、数学、常识推理等任务上的表现。

**核心思想**：
不要求模型直接“猜”答案，而是让它像人一样，一步一步地思考，并把思考过程写出来。这样不仅能提高准确率，还能让推理过程可解释、可检查。

**特点**：

- **显式推理**：模型输出中间步骤，而不仅仅是最终结果。
- **分解问题**：将复杂任务拆分为多个易于处理的子任务。
- **增强性能**：对于多步逻辑、算术、符号推理等任务，CoT 能显著提升正确率。
- **可解释性**：用户可以看到模型的“思路”，便于调试或信任判断。

**简单示例**（数学问题）：

> 问：一个蛋糕分成8等份，小明吃了2块，小红吃了3块，他们一共吃了多少块？还剩多少块？
> 思维链输出：
>
> 1. 总共8块。
> 2. 小明吃2块，小红吃3块，一共吃了 2+3=5块。
> 3. 剩下 8-5=3块。
>    答：一共吃了5块，还剩3块。

**与伪代码的区别**：
伪代码用于**描述算法逻辑**，思维链用于**模型显式推理过程**。思维链更贴近自然语言，常用于提升AI的推理能力。
